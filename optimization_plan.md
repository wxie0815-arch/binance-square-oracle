# 币安广场流量预言机 L7 文章生成优化思路方案

通过对 `binance-square-oracle` 仓库核心代码的深入分析，我发现 L7 文章生成部分确实存在流程繁琐、数据冗余和 API 响应时间过长的问题。以下是针对这些瓶颈的详细分析与优化思路。

## 一、 核心瓶颈分析

### 1. 数据获取冗余与过度截断
在当前架构中，数据获取存在严重的重复和浪费：
- **重复获取**：`oracle_main.py` 在调用 L7 之前已经通过 `ContentCombo` 获取了 `skills_data`，但 `L7_article_generator.py` 内部的 `generate_article_v3` 又调用了一次 `_get_combo_data`，导致完全相同的 8-19 个 HTTP 请求被执行了两次。
- **层级重叠**：L0-L4 层已经获取了大量数据（如 L1 的 `social_hype` 和 `trending_tokens`，L2 的 `smart_money_signals`），而 `ContentCombo` 又重复获取了这些数据。
- **无效传输**：收集到的海量数据（包括 L0 的 2000+ 条帖子、L5 的融合报告、Skills Hub 的深度数据）最终被合并为一个巨大的 JSON 字符串，但在 `writing_skill.py` 中被强行截断：`market_data_str[:2000]` 和 `combo_data_str[:3000]`。这意味着耗时获取的 90% 以上的数据根本没有进入 LLM 的上下文。

### 2. 三阶段写作流程耗时过长
当前的写作流程分为三个独立的 LLM 调用阶段：
1. **writing-plans**：生成结构化 JSON 计划。
2. **copywriting**：根据计划生成初稿。
3. **humanizer-zh**：对初稿进行去 AI 味处理。

这种串行设计导致了显著的延迟：
- 每次调用都需要等待 LLM 响应，总耗时是单次调用的 3 倍以上。
- 阶段 2 和阶段 3 传递的上下文越来越大（阶段 2 传递完整 JSON 计划，阶段 3 传递完整初稿），增加了 Token 消耗和处理时间。
- `humanizer-zh` 作为一个独立的 Skill，其 28 条规则和复杂的 Prompt 增加了 LLM 的推理负担。

## 二、 优化思路方案

为了解决上述问题，建议从“数据精简”和“流程整合”两个维度进行优化。

### 思路一：建立信息筛选机制（数据瘦身）

**目标**：消除重复请求，提取核心指标，避免 LLM 上下文截断。

1. **消除重复调用**：
   - 移除 `L7_article_generator.py` 中的 `_get_combo_data` 调用，直接使用 `oracle_main.py` 传入的 `skills_data`。
   - 梳理 L0-L4 与 `ContentCombo` 的重叠部分，如果 L1/L2 已经获取了数据，L7 阶段就不再重复请求。

2. **构建“核心摘要”数据结构**：
   - 不要将原始的 API 响应 JSON 直接塞给 LLM。
   - 在传入 L7 之前，编写一个数据清洗函数，提取最关键的指标。例如：
     - **情绪**：只保留 `fused_sentiment` 的结论（如“偏多，建议发布行情分析”）。
     - **代币**：只保留 `fused_coins` 的 Top 3，包含名称和核心异动原因。
     - **话题**：只保留 `fused_topics` 的 Top 3。
   - 将清洗后的精简文本（控制在 1000 字符以内）传递给 LLM，彻底废除 `[:2000]` 这种粗暴的截断方式。

### 思路二：整合三阶段写作流程（流程降维）

**目标**：将 3 次 LLM 调用合并为 1-2 次，大幅缩短生成时间。

#### 方案 A：二阶段整合（推荐，平衡质量与速度）
将 `writing-plans` 和 `copywriting` 合并为一步，保留 `humanizer-zh` 作为独立的润色步骤。
- **阶段 1（大纲+初稿）**：给 LLM 提供精简后的数据和风格指纹，要求其先在内部思考（或输出简短的提纲），然后直接输出文章初稿。
- **阶段 2（去 AI 味）**：将初稿交给 `humanizer-zh` 进行最终优化。
- **优势**：减少了一次长文本生成的等待时间，同时保留了专门的去 AI 味机制，确保文章质量。

#### 方案 B：一阶段全能（极致速度）
将三个阶段的 Prompt 核心逻辑提炼，合并为一个强大的 System Prompt。
- **Prompt 重构**：
  - 角色设定：你是一个深谙币安广场爆款逻辑的加密分析师。
  - 任务：根据提供的核心数据，直接写出一篇符合特定风格的文章。
  - 约束（融合 Humanizer 规则）：必须使用短句，禁用“赋能、底层逻辑”等词汇，多用“说实话、其实”等口语，代币格式必须为 `$BTC`。
- **优势**：只需 1 次 LLM 调用，速度最快。
- **挑战**：对 LLM 的指令遵循能力要求极高，可能偶尔会出现“AI 味”残留。建议配合能力较强的模型（如 `gpt-4o` 或 `claude-3.5-sonnet`）使用。

### 思路三：引入流式输出与异步处理

1. **流式输出 (Streaming)**：
   - 如果最终选择保留多阶段或长文本生成，建议在 `writing_skill.py` 中启用 LLM 的 `stream=True`。
   - 虽然总耗时不变，但可以极大地提升用户的体感响应速度。
2. **并行数据获取**：
   - 确保 `oracle_main.py` 中的 L0-L4 层数据获取和 `ContentCombo` 数据获取是完全异步并行的，减少前置数据准备的时间。

## 三、 总结建议

当前 L7 繁琐的根本原因在于**“过度收集数据”**和**“过度细分步骤”**。

最立竿见影的优化路径是：
1. **拦截重复 API**：修复 `oracle_main.py` 和 `L7` 之间重复获取 `combo_data` 的 Bug。
2. **数据摘要化**：用一个 500 字的“核心情报简报”代替几万字的原始 JSON。
3. **合并写作阶段**：将 `writing-plans` 和 `copywriting` 合并，直接生成初稿，再进行一次 `humanizer` 润色。

这样修改后，API 请求量将减少一半，LLM 调用次数减少 1/3，整体 L7 耗时预计可缩短 40%-50%。
