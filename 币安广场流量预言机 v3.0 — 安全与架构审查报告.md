| `data_6551.py` | `BASE` | `"https://ai.6551.io"` | 已改为 `os.environ.get()` |

**发现 2**：6551 API 的 Base URL（`https://ai.6551.io`）在多个历史提交中被硬编码，虽然已在 `9192b7a` 中修复为环境变量，但历史记录中仍可见。

**建议**：

2. 考虑使用 `git filter-branch` 或 `BFG Repo Cleaner` 清理 Git 历史中的敏感信息，或者直接重建仓库（删除旧仓库，新建一个只包含最新代码的仓库）。
3. Telegram Chat ID `1474765089` 仍然暴露在 Git 历史中，虽然 Chat ID 本身不是高危信息，但结合 Bot Token 可被利用。

### 2.2 `__pycache__` 被提交到 Git（严重度：低）

Git 历史中发现了 10 个 `.pyc` 文件被提交。虽然 `.gitignore` 已经包含了 `__pycache__/`，但这些文件是在 `.gitignore` 添加之前被提交的。

**建议**：执行以下命令彻底清理：
```bash
git rm -r --cached __pycache__/
git commit -m "chore: remove cached pyc files"
```

### 2.3 `.env` 加载代码重复且存在注入风险（严重度：低）

几乎每个 `L*.py` 文件的头部都有相同的 `.env` 加载代码：
```python
_env = _pl.Path(__file__).parent / ".env"
if _env.exists():
    for _line in _env.read_text().splitlines():
        if _line.strip() and not _line.startswith("#") and "=" in _line:
            _k, _v = _line.split("=", 1)
            import os as _os; _os.environ.setdefault(_k.strip(), _v.strip())
```

**问题**：
1. 代码重复了 6 次以上，违反 DRY 原则。
2. 这个简易解析器不处理引号包裹的值（如 `KEY="value with spaces"`），可能导致值解析错误。
3. 如果 `.env` 文件被恶意修改（如注入 `PATH=/malicious/path`），可能影响系统行为。

**建议**：使用成熟的 `python-dotenv` 库替代手写解析器，并将加载逻辑集中到一个 `config.py` 中：
```python
# config.py
from dotenv import load_dotenv
load_dotenv()
```

### 2.4 `subprocess` 调用的安全性（严重度：低）

`L6_style_analyzer.py`、`L6_article_generator.py` 和 `post_with_tokens.py` 中使用了 `subprocess.run` 来调用外部脚本。虽然当前参数是受控的，但 `username` 参数直接拼接到命令行中，如果未来接受用户输入，可能存在命令注入风险。

**建议**：将 `binance_profile_analyzer.py` 的核心函数改为 Python 模块导入方式调用，而非通过 `subprocess` 启动新进程。

### 2.5 HTTP 请求缺少超时和重试统一管理（严重度：低）

各层的 HTTP 请求超时时间不一致（10s、12s、15s、20s、30s、35s），且没有统一的重试机制。

**建议**：创建一个统一的 HTTP 客户端工具函数，内置指数退避重试和统一超时配置。

## 三、L7 文章生成层深度优化建议

L7 是用户特别关注的层，当前实现已经相当成熟（L5 数据注入 + L6 风格指纹锁定 + API Failover），以下是进一步的优化方向：

### 3.1 引入"事实校验层" (Fact-Checking Guard)

**问题**：LLM 生成的文章可能包含"幻觉数据"——捏造不存在的价格、涨跌幅或时间点。在币安广场这样的金融内容平台上，错误的数据会严重损害账号信誉。

**改进方案**：在 `post_process()` 之后增加一个 `fact_check()` 函数：

```python
def fact_check(article: str, fusion_report: dict) -> tuple[bool, str]:
    """校验文章中的数字是否与原始数据一致"""
    import re
    
    # 提取文章中的价格数字
    prices = re.findall(r'\$?(\d{1,6}(?:,\d{3})*(?:\.\d+)?)', article)
    
    # 提取文章中的百分比
    percentages = re.findall(r'(\d+(?:\.\d+)?)\s*%', article)
    
    # 与 fusion_report 中的真实数据比对
    real_btc = fusion_report.get("fused_coins", [{}])[0].get("details", {}).get("current_price")
    
    for p in prices:
        p_val = float(p.replace(",", ""))
        if real_btc and abs(p_val - real_btc) / real_btc > 0.1:  # 偏差超过10%
            return False, f"价格数据异常: 文章中{p_val} vs 真实{real_btc}"
    
    return True, "ok"
```

### 3.2 动态结构模板 (Dynamic Structure Templates)

**问题**：当前 L7 的文章结构是硬编码的两种模式（短文 ≤150 字 / 长文），无法根据市场状态动态调整叙事结构。

**改进方案**：定义 4-5 种结构模板，由 L5 的情绪评分自动选择：

| 情绪状态 | 结构模板 | 适用场景 |
|----------|---------|---------|
| 极度贪婪 (≥70) | **警告体**：异常数据 → 历史对比 → 风险提示 → 反问 | 市场过热时的逆向内容 |
| 偏多 (55-69) | **分析体**：数据钩子 → 趋势解读 → 关键位分析 → 操作思路 | 常规行情分析 |
| 中性 (45-54) | **教育体**：概念引入 → 数据佐证 → 实操方法 → 互动提问 | 科普和策略分享 |
| 偏空 (30-44) | **心态体**：市场现状 → 情绪描写 → 理性分析 → 鼓励式结尾 | 下跌行情的心态管理 |
| 极度恐惧 (<30) | **逆向体**：恐慌数据 → 智能钱异动 → 历史底部对比 → 悬念 | 恐慌行情的逆向思维 |

### 3.3 多轮生成与自评 (Generate-then-Evaluate)

**问题**：当前 L7 只调用一次 LLM 生成文章。LLM 的输出质量存在随机性，单次生成可能不是最优结果。

**改进方案**：
1. 调用 LLM 生成 2-3 篇候选文章（可以使用更低成本的模型如 `gpt-4.1-mini`）。
2. 使用一个轻量级的评分 Prompt 对候选文章进行打分（风格匹配度、数据准确性、互动潜力）。
3. 选择得分最高的文章作为最终输出。

这种方式虽然增加了 API 调用次数，但可以显著提升文章质量的下限。

### 3.4 本地模板降级 (Template Fallback)

**问题**：当所有 LLM API 都不可用时，L7 返回空字符串，导致整个预言机无法产出内容。

**改进方案**：在所有 API 失败后，使用 Python 字符串模板直接拼接一篇基础行情播报：

```python
FALLBACK_TEMPLATES = [
    "${{coin}} 在{{price}}附近震荡，24小时{{direction}}{{change}}%。"
    "链上数据显示智能钱{{smart_action}}，恐惧贪婪指数{{fgi}}。"
    "这个位置你怎么看？",
    
    "{{coin}}今天{{direction}}了{{change}}%，但我注意到一个数据——"
    "智能钱{{smart_action}}。上次出现这种情况是{{days}}天前，"
    "当时的走势你还记得吗？",
]
```

### 3.5 Prompt 精简与 Token 优化

**问题**：当前 L7 的 Prompt 非常长（风格指纹 + 写作法则 + 互动规律 + 市场数据 + 规则），可能超过 2000 tokens，导致 LLM 的注意力分散。

**改进方案**：
1. 将"强制规则"部分移到 `system` message 中，与 `user` message 分离。
2. 对 `style_fingerprint` 做摘要压缩，只保留最关键的 5 条规则。
3. 将 `patterns` 和 `mastery` 的内容做去重和精简，控制在 300 tokens 以内。

## 四、整体架构改进建议

### 4.1 统一配置中心 (`config.py`)

**问题**：每个模块都有重复的 `.env` 加载代码、硬编码的路径（如 `/home/ubuntu/.openclaw/workspace`）和常量定义。

**改进方案**：创建一个 `config.py` 作为配置中心：

```python
# config.py
import os
from dotenv import load_dotenv
load_dotenv()

WORKSPACE = os.environ.get("WORKSPACE", "/home/ubuntu/.openclaw/workspace")
MEMORY_DIR = f"{WORKSPACE}/memory"

# API Keys
PUCODE_API_KEY = os.environ.get("PUCODE_API_KEY", "")
SQUARE_API_KEY = os.environ.get("SQUARE_API_KEY", "")

# HTTP 配置

```

### 4.2 异步并发数据采集

**问题**：`oracle_main.py` 串行执行 L0-L4，总耗时是各层耗时之和。

**改进方案**：使用 `concurrent.futures.ThreadPoolExecutor` 实现并发（比 asyncio 改动更小）：

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def run_oracle_parallel():
    with ThreadPoolExecutor(max_workers=5) as executor:
        futures = {
            executor.submit(run_l0): "L0",
            executor.submit(run_l1): "L1",
            executor.submit(run_l2): "L2",
            executor.submit(run_l3): "L3",
            executor.submit(run_l4): "L4",
        }
        reports = {}
        for future in as_completed(futures):
            layer = futures[future]
            try:
                reports[layer] = future.result()
            except Exception as e:
                reports[layer] = empty_report(layer)
    return reports
```

这样可以将数据采集时间从 ~15s 压缩到 ~3s。

### 4.3 数据持久化升级

**问题**：`post_history.json`、`style_weights.json` 等 JSON 文件在高频读写时可能出现竞争条件。

**改进方案**：引入 SQLite（Python 内置，零依赖）：

```python
import sqlite3

def init_db():
    conn = sqlite3.connect(f"{MEMORY_DIR}/oracle.db")
    conn.execute("""CREATE TABLE IF NOT EXISTS post_history (
        id INTEGER PRIMARY KEY,
        ts TEXT, key TEXT, post_id TEXT, preview TEXT,
        content_type TEXT, token TEXT,
        view INTEGER DEFAULT 0, like INTEGER DEFAULT 0,
        comment INTEGER DEFAULT 0, share INTEGER DEFAULT 0,
        checked BOOLEAN DEFAULT 0
    )""")
    return conn
```

当前 `quality_check()` 只检查字数、emoji 和 7 个禁用词。

3. **代币格式校验**：确保所有代币名称都使用了 `$BTC` 格式。
